# 信贷模型论文伴读与网页展示二次改版规格

## 1. 目标

在不改变已验证风险模型和信贷优化结果的前提下，完成两项展示层改进：

1. 将当前约11页的论文扩写为18至22页，使没有统计、机器学习或运筹优化基础的读者也能沿着完整逻辑理解建模过程。
2. 将当前结果型Dashboard改造成“论文伴读 + 结果展示”页面，既能快速核查结果，也能解释数据如何转化为信贷策略。

本轮不重新选择模型，不更改PD阈值、预算、LGD、资金成本、禁贷规则和疫情冲击系数。任何模型级调整必须另行提出和验证。

## 2. 受众与表达原则

### 2.1 主要受众

- 第一次接触数学建模的学生。
- 需要快速检查逻辑完整性的数学建模评委。
- 希望复算信贷策略的技术读者。

### 2.2 统一解释顺序

每个主要模型按以下顺序展开：

1. 为什么需要这个模型。
2. 用一个简单数字例子建立直觉。
3. 定义变量和数学公式。
4. 解释程序如何求解。
5. 展示真实数据结果。
6. 说明银行应如何理解结果，并指出假设和局限。

不以堆叠公式代替解释，也不把通俗解释与数学推导拆成互不相连的两套文本。

## 3. 论文设计

### 3.1 篇幅目标

最终PDF控制在18至22页，目标约20页。建议分配如下：

| 内容 | 目标页数 |
|---|---:|
| 摘要、问题重述与问题分析 | 2 |
| 数据口径、假设、符号与总流程 | 3 |
| 特征工程 | 2 |
| 监督PD模型与样本外验证 | 4 |
| 固定尺度TOPSIS辅助解释 | 1至2 |
| 流失率、期望收益与额度优化 | 4 |
| 疫情冲击 | 2 |
| 灵敏度、模型评价与结论 | 2至3 |

页数是验收边界，不通过缩小字号、压缩行距或裁切图表强行满足。

### 3.2 章节结构

1. 问题重述与任务拆解。
2. 问题分析与整体建模路线。
3. 数据理解、清洗规则与基本假设。
4. 符号说明和模型输入输出关系。
5. 发票净额、连续月历、年化与经营特征。
6. 正则逻辑回归PD模型。
7. 类别平衡、先验校正与嵌套重复交叉验证。
8. 固定PD等级和模型验证结果。
9. TOPSIS辅助经营质量解释。
10. 保序流失率模型和利率选择。
11. 单位额度期望收益与收益拆分。
12. 混合整数线性规划和三问策略结果。
13. 疫情赔率冲击、重新评级与重新优化。
14. 灵敏度分析、优点、局限和扩展方向。
15. 结论。

### 3.3 必须新增的教学内容

- 完整建模流程图，清楚标出每一阶段的输入和输出。
- 符号说明表，至少覆盖企业索引、PD、流失率、留存率、利率、额度、LGD、资金成本、预算和选择变量。
- 发票算例：展示正常发票、负数红冲和作废票据如何影响净额。
- 连续月历算例：解释无交易月份为什么必须补0。
- 逻辑回归算例：用一至两个特征演示线性得分如何转成0至1的概率。
- 交叉验证示意：说明训练内效果和样本外效果的区别，以及内外层分别做什么。
- 指标白话解释：ROC-AUC、PR-AUC、Brier和Log Loss分别回答什么问题。
- TOPSIS角色说明：它评价经营质量，但不再代替违约概率。
- 流失率定价算例：展示提高利率同时增加利息和客户流失的两面性。
- 期望收益拆分：利息收入、违约损失、资金成本和客户留存的对应关系。
- 小规模额度分配算例：用少量虚拟企业解释MILP的目标和约束，不把虚拟数据混入正式结果。
- 疫情赔率算例：说明为什么调整赔率比直接乘概率更合理。
- 每个结果表后增加“结果意味着什么”，连接统计结果和银行决策。

### 3.4 动态数字要求

论文中的模型效果、等级分布、放贷数量、预算、平均利率、收益、疫情变化数量和灵敏度指标继续来自 `generated_metrics.tex`。新增结果数字时，应先扩展统一指标文件和LaTeX宏，不得直接把运行结果手写进正文。

教学算例必须明确标注为“示例”，不得与附件中的真实企业混淆。

## 4. 网页设计

### 4.1 定位与设计语言

页面类型为“可审查的论文伴读页”，而不是营销落地页，也不是高交互产品后台。

Design Read：面向数学建模初学者和评委的教学型分析页面，严谨、克制、偏学术编辑风格。

设计参数：

- `DESIGN_VARIANCE: 4`
- `MOTION_INTENSITY: 2`
- `VISUAL_DENSITY: 7`

前端采用现有原生HTML/CSS生成方式，不引入React、打包器、CDN或在线字体。保留完全离线打开能力。

### 4.2 信息架构

1. 模型全景。
2. 数据如何处理。
3. 风险如何预测。
4. TOPSIS的辅助角色。
5. 利率如何确定。
6. 额度如何分配。
7. 疫情如何调整。
8. 模型是否可靠。
9. 参数、假设与结果文件。

保留现有 `validation`、`strategy`、`emergency`、`sensitivity` 锚点，在此基础上增加新锚点，避免破坏已有导航定位。

### 4.3 页面结构

- 顶部导航保持单行，桌面端高度不超过72像素。
- 首屏左侧呈现模型目标和一句话结论，右侧呈现真实的模型流程关系，不放版本号装饰或虚构界面截图。
- 首屏之后展示核心验证指标和三问策略摘要。
- 桌面端正文采用主内容加辅助栏结构。辅助栏显示本节要点、术语或参数；移动端折叠为单栏。
- 主要内容使用留白、分隔线、表格和不同宽度的图文组合建立层次，避免连续堆叠相同卡片。
- 复杂解释使用原生 `details/summary` 折叠组件，默认正文仍能形成完整叙事。
- 公式使用离线HTML文本或本地生成图片，不依赖MathJax CDN。
- 数字采用等宽数字字体回退，中文正文使用系统中文字体。

### 4.4 视觉规范

- 全页锁定浅色主题。
- 深海军蓝用于导航和主要标题，青绿色为唯一强调色。
- 统一使用12像素左右的内容容器圆角；按钮或折叠控件允许较小圆角，但不得混用大量胶囊形状。
- 卡片只用于确实需要边界或层级的内容，普通解释以留白和分隔线组织。
- 不使用渐变文字、玻璃拟态、外发光、装饰性状态点、滚动提示和无意义动画。
- 页面可使用轻量悬停、焦点和展开反馈，并遵守 `prefers-reduced-motion`。
- 所有文字与背景达到WCAG AA可读对比度。
- 不使用长破折号或半长破折号字符。

### 4.5 新增内容和图表

- 数据规模与数据来源概览。
- 建模流程及各阶段输入输出关系。
- 16项PD特征的分组解释。
- 样本外验证指标的通俗含义。
- 利率、客户流失和单位收益的关系说明。
- 期望收益构成图。
- 三问策略对比。
- 疫情调整前后的PD、等级和额度变化。
- TOPSIS灵敏度和指标移除结果。
- 参数假设、适用边界和结果文件入口。

所有统计图使用真实运行结果，不生成装饰性假数据。

## 5. 生成架构

统一数据流如下：

```text
main.py
  -> report_metrics.json
  -> generated_metrics.tex
  -> output/charts/*.png
  -> paper/paper.tex -> paper/paper.pdf
  -> reporting.py -> dashboard.html
```

实现职责：

- `paper/paper.tex`：教学型正文、公式、表格和图表布局。
- `reporting.py`：统一指标、LaTeX宏和网页模板。
- `gen_paper_charts.py`：所有论文和网页共用的真实统计图。
- `tests/test_model.py`：模型约束和展示产物完整性检查。

本轮不把论文正文自动生成为网页，二者共享指标、图表和解释顺序，但分别针对打印阅读与屏幕阅读组织版式。

## 6. 异常与退化处理

- 缺失统一指标文件时，主流程应先生成，不允许网页静默显示占位数字。
- 图表生成失败时，整体运行返回错误，不输出引用缺失图片的最终网页。
- 网页生成后检查本地图片路径、锚点目标和外部链接。
- LaTeX宏缺失时应在编译阶段失败，不使用手写默认结果掩盖问题。
- 小屏表格使用水平滚动容器，正文不能被表格撑出视口。
- 打印时隐藏吸附导航，确保正文、表格和图表保持清晰。

## 7. 验收标准

### 7.1 论文

- 最终PDF为18至22页。
- XeLaTeX编译零错误，无Overfull或Underfull警告。
- 每个主要模型都包含直觉、公式、求解、真实结果和结果解读。
- 教学示例与真实数据有明确标识。
- 所有动态结果与 `report_metrics.json` 一致。
- 抽查首页、总流程、PD模型、额度优化和结论页面。

### 7.2 网页

- 完全离线，无CDN、外部字体和远程脚本。
- 桌面和移动布局均可阅读，导航、表格和折叠面板可用。
- 内容覆盖九个信息模块，并保留四个原锚点。
- 颜色、圆角和字体规范一致，只有一个强调色。
- 不包含长破折号、模拟精确数字或装饰性假图。
- 所有图片路径、导航锚点和可访问文本有效。

### 7.3 回归验证

- 原有模型自动测试全部通过。
- 新增论文结构、页数、LaTeX警告、网页章节、外链、锚点和本地资源测试。
- Excel预算、D级禁贷和公式检查继续通过。

## 8. 明确不做

- 不更换监督PD模型。
- 不改变TOPSIS、流失率、MILP和疫情冲击的数学定义。
- 不引入在线服务、Web框架、用户登录或数据库后端。
- 不制作与真实模型无关的营销插画或复杂动画。
- 不为了增加篇幅重复同一结论或大段复制代码。
